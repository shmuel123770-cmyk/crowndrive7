rev.33 (2026-07-13) — THE stuck root cause (CSP) + auto-cache-recovery + full make/model list:
- *** THE REAL "loads for an hour" *** — found in the user's DevTools: dozens of
  `firebase-database-compat` long-poll requests (`.lp?start=…`) were `blocked:CSP`. The CSP allowed
  the database in connect-src (WebSocket) but script-src only had 'self' + gstatic — so on any
  network where WebSockets don't work, Firebase's long-poll fallback (loaded as <script>) was blocked
  and the database NEVER connected → the personal area hung forever. FIX (_headers): script-src +
  connect-src now include https://*.firebaseio.com and https://*.firebasedatabase.app (+ wss). This
  is device/network-dependent, which is exactly why it hit "some users".
- NO MORE MANUAL CACHE CLEARING: the service worker, on activate, now force-reloads any open page
  still running old code (client.navigate) — so stale users recover automatically, no "clear cache".
- CAR MAKES/MODELS: expanded to 49 manufacturers (Acura…Volvo) with full model lists, ALWAYS sorted
  A→Z (makes + models). "אחר…" still lets you type any make/model not in the list.
- Changed: _headers, sw.js, js/views.js (+ rev.32: role client-write, render anti-starvation).
  Re-publish the Firebase rules (rev.32 role rule). Assets ?v=26.

rev.30 (2026-07-13) — THE real "loads for an hour" fix + cache-free service worker:
- ROOT CAUSE (found by inspection, verified in-browser): firebase.js had a TOP-LEVEL `await` on
  auth.setPersistence(). A top-level await blocks the ENTIRE module graph — so if that call HANGS
  (slow/locked/blocked IndexedDB on some devices), no module ever finishes loading and the app is
  stuck on the loading screen indefinitely. Earlier fixes only caught it *throwing*, not *hanging*.
  Now setPersistence is fired WITHOUT await (Firebase's default is already LOCAL, so the login still
  persists per-browser). There is now NO top-level await anywhere in the module graph → init can
  never block. Verified: fresh load renders HOME, 0 caches.
- SERVICE WORKER IS NOW CACHE-FREE (per "keep only the login in cache"): it stores NOTHING — no
  HTML/JS/CSS — and wipes every old cache on activate. Every request goes straight to the network,
  so the site can never be stuck on an old version. The login is kept by Firebase in IndexedDB (not
  the SW), so users stay signed in in the same browser. index.html still auto-reloads once when a
  new worker takes over, so anyone stuck on an old cached bundle is freed automatically. Assets ?v=25.

rev.28 (2026-07-13) — universal stability + responsive (per the work-instructions doc):
STABILITY (the "stuck on certain devices for an hour"):
- firebase.js: the top-level `await auth.setPersistence(LOCAL)` is now wrapped in try/catch. On
  browsers that block storage (Safari Private, some in-app browsers) it used to REJECT and crash the
  whole module → blank/stuck screen. Now it degrades gracefully and the site loads. (LOCAL kept =
  session persists per-browser, not per-device — your request.)
- auth.js: the email-verification sync (reload + getIdToken + profile-save) moved OFF the critical
  path. It sat behind resolveReady(), so a hanging function froze auth "readiness" forever. Now auth
  is marked ready immediately; the sync runs in the background.
- app.js: 10s safety net — if Firebase never reports auth state, authSettled is forced so gated
  screens resolve instead of spinning.
- api.js: every server fetch is time-boxed to 25s (AbortController) with a clean Hebrew error.
- sw.js: network-first now has a 6s cap → falls back to cache/app-shell instead of hanging on a
  stalled network. Cache bumped to v13.
RESPONSIVE (all breakpoints / iPhone / Android / weak devices):
- Modals + chat use dvh (with vh fallback) so the mobile keyboard / URL-bar can't clip them.
- Inputs forced to 16px on phones/tablets → no iOS auto-zoom on focus.
- Sticky header respects the iPhone notch at every breakpoint (safe-area-inset-top).
- prefers-reduced-motion drops animations; very small / slow devices drop heavy backdrop-blur.
- (Already present and kept: overflow-x:hidden, max-width:100%, safe-area bottom bar, dvh chat.)
Verified in-browser at 375px: no horizontal scroll, inputs 16px, notch padding applies, clean boot,
no console errors. Assets ?v=24. Only presentation/stability files changed — NO Firebase Rules

rev.25 (2026-07-13) — personal area can no longer hang on "טוען את האזור האישי…":
- ROOT CAUSE: in store.startPrivate the admin-flag read `admins/<uid>.once()` ran BEFORE the profile
  listener was attached. If that read threw (denied / expired token) or stalled, startPrivate crashed
  after setting store.user but before the profile listener — so store.profileLoaded stayed false and
  dashboard() spun forever. Fixes: (1) the admin read is wrapped in try/catch + a 6s timeout → never
  crashes/stalls the flow; (2) the profile listener sets profileLoaded=true even on error; (3) an 8s
  safety net flips profileLoaded so the UI always falls through (to the correct dashboard or the
  complete-profile screen) instead of an endless spinner. Only js/store.js changed (network-first).

rev.20 (2026-07-12) — ADMIN CAN DELETE cars + users (was blocked):

  which chokes when Netlify delivers the body base64-encoded (the same failure that broke
  publishing). This made admin-action (user-delete/block/update/car-owner) and others fail.
  FIX: every function now uses the base64-safe parseBody(event) + a clean 400 on a bad body.
- CAR DELETE is now a CLIENT-WRITE (like publishing) — deleteCar() writes cars/<id> +
  privateCarDetails/<id> to null directly, so it works even if the function misbehaves. New DB
  rules let the car's owner OR an admin delete a car (and its private details); admins can delete
  ANY car. Owners are still blocked from deleting a car with an open booking (admin bypasses).
- USER DELETE now goes through the fixed admin-action (functions are live + admin is recognised;
  the base64 parse was the blocker). MUST publish the updated DB rules + deploy the functions.
  Assets ?v=22.
